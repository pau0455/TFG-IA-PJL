Desarrolla con AITalk

Arquitectura del Sistema IA Agéntica

AITalk es un sistema automatizado e inteligente de atención al cliente diseñado para cubrir las necesidades de empresas de hostelería, basado en una arquitectura de IA agéntica. El objetivo del proyecto es crear un asistente virtual capaz de gestionar reservas, responder consultas sobre el menú y alérgenos, y mantener una conversación fluida en lenguaje natural a través de Telegram.

Se ha seguido una metodología ágil utilizando herramientas de gestión tipo Kanban, específicamente Taiga. El desarrollo se ha dividido en pequeños flujos de trabajo modulares dentro del orquestador n8n. En resumen, se utilizan múltiples herramientas dentro de n8n como Telegram, Google Calendar, PostgreSQL y una API de IA para lograr la creación de un agente.

Contexto y Justificación

El Problema en la Hostelería

El entorno actual de la pequeña y mediana hostelería se caracteriza por la multiplicidad de canales de comunicación. Los clientes intentan contactar con los establecimientos a través de llamadas telefónicas, redes sociales o aplicaciones de mensajería instantánea. Esta dispersión genera un volumen de consultas que el personal del restaurante, enfocado en la operativa física del local, difícilmente puede gestionar de forma ininterrumpida.

Actualmente, la gestión de reservas en este tipo de negocios oscila entre dos extremos. Por un lado, se encuentran los métodos manuales basados en libros de reservas o anotaciones informales, los cuales son propensos a errores humanos, solapamientos y pérdida de información. Por otro lado, existen plataformas digitales de terceros que centralizan las reservas, pero que suelen implicar el pago de altas comisiones o la pérdida de control sobre los datos de los clientes.

Además, los usuarios demandan cada vez más inmediatez. La imposibilidad de realizar una reserva a altas horas de la noche o la falta de respuesta inmediata ante consultas comunes, como la disponibilidad de un menú específico o la presencia de alérgenos en los platos, se traduce frecuentemente en la pérdida de clientes potenciales.

Por qué AITalk

AITalk nace en este escenario para democratizar el acceso a tecnologías de vanguardia. Al aprovechar herramientas de código abierto y modelos de lenguaje de última generación, se propone una alternativa tecnológica que otorga autonomía al negocio, centraliza la información y proporciona un servicio ininterrumpido sin incrementar los costes de personal.

El desarrollo de este proyecto se justifica por la necesidad imperante de resolver la sobrecarga administrativa y económica que sufren las pequeñas empresas al intentar ofrecer una atención al cliente continua. Delegar estas tareas rutinarias a un agente de inteligencia artificial accesible a través de una plataforma de uso masivo mejora radicalmente la experiencia del usuario. El cliente final no necesita instalar aplicaciones de terceros ni navegar por interfaces web complejas, sino que interactúa con el restaurante de la misma manera que lo haría con un contacto humano.

Justificación Técnica

Desde el punto de vista técnico, la construcción de AITalk supone un desafío integrador que justifica plenamente su realización como proyecto de fin de ciclo. Exige la combinación de múltiples disciplinas informáticas adquiridas durante la formación académica: el despliegue de infraestructuras de red avanzadas, la administración de sistemas gestores de bases de datos relacionales, la programación de lógica de negocio y la integración de interfaces de programación de aplicaciones externas.

La adopción de un orquestador de flujos de trabajo permite demostrar la capacidad de diseñar arquitecturas modulares. Esta modularidad garantiza que el sistema sea tolerante a fallos y fácilmente escalable en el futuro. Por consiguiente, AITalk no solo resuelve un problema comercial tangible, sino que sirve como demostración práctica de competencias avanzadas en administración de sistemas y desarrollo de software moderno.

Objetivos del Proyecto

Objetivo General

Desarrollar un sistema automatizado inteligente capaz de gestionar tareas y conectar diferentes servicios mediante flujos de trabajo modulares, ofreciendo una atención al cliente autónoma y en lenguaje natural.

Objetivos Específicos

  • Integración de IA: Conectar el modelo Devstral latest de Mistral para interpretar intenciones y extraer datos clave. El agente ha sido configurado mediante prompt engineering para comprender el contexto de un restaurante y responder de manera coherente.
  • Interfaz de Usuario: Implementar un bot de Telegram como canal principal de comunicación fluida y accesible para el usuario final, procesando tanto entradas de texto como archivos de voz.
  • Gestión de Reservas: Sincronizar la IA con Google Calendar mediante API para visualizar, crear y modificar eventos en tiempo real, evitando solapamientos y validando disponibilidad de forma autónoma.
  • Gestión de Datos: Diseñar y conectar una base de datos PostgreSQL para almacenar menús, alérgenos, locales e información de clientes, permitiendo a la IA realizar consultas dinámicas y registrar los datos necesarios.

Estrategia y Planificación

El enfoque seleccionado para la consecución del proyecto se basa en un desarrollo de carácter incremental y modular. En lugar de intentar construir un sistema monolítico desde el inicio, se decidió dividir la arquitectura en componentes independientes que interactúan entre sí. Esta decisión estratégica minimiza los riesgos técnicos, ya que permite aislar los errores y facilita la validación de cada pieza antes de su integración en el conjunto global.

El punto de partida del desarrollo consistió en la validación del modelo de lenguaje. Se priorizó comprobar si la inteligencia artificial era capaz de asumir un rol específico, comprender el contexto de un restaurante y responder de manera coherente sin sufrir desviaciones o alucinaciones. Una vez asegurada la viabilidad del razonamiento de la máquina, la estrategia dictó avanzar hacia la conexión de herramientas externas, integrando progresivamente la memoria a largo plazo, la gestión de calendarios y, finalmente, la interfaz del usuario final.

Se identificó como posible dificultad la latencia en las respuestas de los modelos de inteligencia artificial y los límites de peticiones de las interfaces públicas. Para mitigar este riesgo, la estrategia de desarrollo contempló desde el inicio la posibilidad de intercambiar los modelos de lenguaje subyacentes sin alterar la lógica central del flujo de trabajo, asegurando así la continuidad operativa ante posibles cuellos de botella.

Módulo Telegram

Actúa como frontend entre el cliente y el sistema. Escucha peticiones y retorna las respuestas generadas por la IA directamente al chat del usuario en su dispositivo móvil o de escritorio.

Por qué Telegram

Al delegar la tarea de atención al cliente a un agente de IA accesible mediante una plataforma de uso masivo como Telegram, se mejora significativamente la experiencia del usuario final. El cliente no necesita navegar por interfaces web complejas ni sufrir tiempos de espera telefónicos. Basados en el conocimiento y experiencia obtenidos durante la formación, Telegram resultó ser la opción más robusta y versátil para esta tarea frente a otras alternativas de mensajería.

Procesamiento Dual: Texto y Voz

El nodo de Telegram escucha continuamente mensajes nuevos. Cuando llega un mensaje, se pasa a un nodo condicional "if" que tiene como objetivo diferenciar los mensajes de voz de los de texto. Esta lógica permite cumplir con uno de los requisitos secundarios: el soporte de Speech-to-Text. Si el mensaje es de texto, se procesa directamente; si es de voz, se envía primero al módulo de transcripción antes de pasar al agente IA.

Requisitos Funcionales

Los requisitos funcionales definen las capacidades concretas que el sistema AITalk debe cumplir para satisfacer las necesidades de atención al cliente en el sector de la hostelería.

  • RF1 - Interfaz Telegram: El sistema debe recibir y enviar mensajes a los usuarios a través de un bot de Telegram, funcionando como el canal principal de comunicación bidireccional entre el cliente y el agente de IA.
  • RF2 - Procesamiento IA: El sistema utilizará la API de Devstral latest para interpretar solicitudes complejas de los usuarios con precisión en formato texto.
  • RF3 - Gestión de Calendario: El sistema permitirá a la IA programar eventos mediante Google Calendar, indicando hora de inicio, finalización y título.
  • RF4 - Consultas BBDD: La IA debe tener acceso de lectura/escritura a una base de datos PostgreSQL para informar sobre menús, alérgenos y registrar los datos necesarios.
  • RF5 - Memoria Conversacional: El sistema debe recordar el contexto de la conversación durante un periodo de tiempo limitado para dar continuidad a las interacciones.

Requisitos No Funcionales

Los requisitos no funcionales establecen las cualidades de calidad que debe cumplir la arquitectura y el diseño del sistema para garantizar su mantenibilidad, usabilidad y evolución futura.

  • RNF1 - Modularidad: La arquitectura basada en n8n debe permitir añadir nuevos módulos o herramientas sin afectar al funcionamiento del sistema completo. Cada nodo realiza una tarea específica y puede ser modificado o reemplazado de forma independiente, lo que garantiza la tolerancia a fallos y la escalabilidad.
  • RNF2 - Naturalidad: El estilo de respuesta del agente debe ser fluido, natural y útil para el usuario final. Se ha trabajado extensamente en el prompt engineering del sistema para que el agente se comporte como un recepcionista profesional, resistente a intentos de desviación del tema o Jailbreak.
  • RNF3 - Escalabilidad: La base de datos y el flujo de trabajo deben estar estructurados de manera que permitan fácilmente añadir nuevas entidades en el futuro. El esquema relacional soporta la incorporación de nuevos locales, menús y servicios sin requerir una reestructuración completa del sistema.

Tareas de Investigación

Durante la fase de planificación del proyecto, se identificaron una serie de líneas de investigación críticas cuyo resultado condicionó las decisiones de diseño y tecnología adoptadas.

  • Análisis comparativo de orquestadores: Se evaluó n8n frente a otras alternativas del mercado para asegurar que cumple con los requisitos de autoalojamiento en local y posee nodos avanzados para el manejo de agentes de IA sin depender de altas cuotas de pago. n8n fue seleccionado por ser de código abierto, con capacidad de alojamiento local y potencia en el manejo de flujos de IA.
  • Investigación de modelos LLM: Se evaluaron y compararon la API de Gemini 3 Flash Preview y Mistral para medir la velocidad de respuesta, la comprensión del lenguaje natural, el coste por token y los límites de la cuota gratuita bajo concurrencia. Devstral latest resultó ofrecer mayor disponibilidad, menor latencia y una tasa de errores casi nula.
  • Diseño del esquema de base de datos: Se determinó la estructura relacional óptima en PostgreSQL para almacenar menús, alérgenos, disponibilidad de mesas y, especialmente, la gestión de la "memoria" o contexto conversacional a largo plazo de la IA. El esquema evolucionó iterativamente durante las fases de prueba.
  • Investigación de la API de Google Calendar: Se analizó la documentación de Google Cloud Console para implementar las operaciones de lectura "get events" y escritura "create events" desde los flujos de n8n de manera segura.
  • Pruebas de Prompt Engineering y seguridad: Se diseñó, probó y refinó el system prompt del agente para que asuma estrictamente su rol como recepcionista del restaurante, sepa cómo invocar las herramientas y sea resistente a intentos de Jailbreak.

Orquestación con n8n

n8n se utiliza como orquestador principal del proyecto, permitiendo integrar y coordinar los diferentes servicios que forman parte de la solución. A través de flujos de trabajo automatizados, n8n gestiona la comunicación entre la inteligencia artificial, la base de datos, los servicios externos como Google Calendar y el servicio de mensajería, en este caso Telegram.

Arquitectura Modular

La arquitectura sigue un modelo de nodos modulares centralizados en n8n. El sistema se divide en piezas más pequeñas que, al unirse, conforman un sistema robusto:

  • Trigger: El nodo de Telegram escucha continuamente mensajes nuevos.
  • Agente IA: El mensaje se pasa al nodo de "if" con el objetivo de diferenciar mensajes de voz de los de texto, para posteriormente ser procesado por el nodo "AI Agent" configurado con el modelo Devstral latest y un rol específico predefinido mediante prompt engineering.
  • Memoria en Postgres: Se utiliza un módulo de memoria permanente almacenando en la propia base de datos PostgreSQL, lo que permite que la IA recuerde el contexto a largo plazo.
  • Tools: Si el agente detecta que necesita accionar algo, llama de forma transparente a los submódulos.
  • Respuesta: El agente formula una respuesta natural y lleva a cabo las acciones necesarias para cumplir la petición del usuario usando los diferentes módulos. n8n la devuelve al usuario a través del bot de Telegram.

Por qué n8n y no otras alternativas

Se ha escogido n8n frente a alternativas comerciales por ser de código abierto, la capacidad de alojamiento local y potencia en el manejo de flujos de IA sin depender de las altas cuotas de otras plataformas de pago. La arquitectura basada en nodos permite estructurar el flujo de manera modular, donde cada nodo realiza una tarea específica como procesar mensajes, consultar información, almacenar datos o generar respuestas.

Migración del Proveedor de IA

Durante la fase inicial de desarrollo del proyecto, se utilizó como principal proveedor de modelos de inteligencia artificial la API de Gemini, desarrollada por Google. En un primer momento, esta solución parecía ser la opción más adecuada debido a su alto rendimiento y capacidades avanzadas de procesamiento de lenguaje natural.

Sin embargo, a medida que avanzaba el desarrollo y aumentaba la demanda global del servicio externo, se comenzaron a experimentar limitaciones significativas derivadas de la alta carga sobre la API. El elevado número de solicitudes concurrentes generadas por usuarios de todo el mundo provocaba tiempos de respuesta inestables, mayores latencias y, en determinados casos, errores en las peticiones.

Tras identificar este problema, se evaluaron alternativas que permitieran mantener la funcionalidad del sistema con mayor estabilidad y control sobre los recursos. Finalmente, se optó por integrar los modelos de Mistral AI, una empresa europea especializada en modelos de lenguaje. La estrategia de desarrollo ya contemplaba la posibilidad de intercambiar modelos, lo que permitió migrar sin alterar la lógica central del flujo de trabajo.

Base de Datos

Conexión directa a la base de datos relacional del restaurante. Se ha seleccionado PostgreSQL por ser un software Open Source robusto y sin coste de licencia. Además de almacenar el inventario, sirve para guardar el contexto de la IA, centralizando toda la información en un único punto.

Esquema Conceptual Completo

Durante el desarrollo, se diseñó un esquema relacional con las siguientes entidades principales:

  • Comida e Ingredientes: Tablas relacionadas donde un Menú incluye platos que a su vez contienen Ingredientes. Se controlan los Alérgenos, permitiendo a la IA responder consultas sobre restricciones alimentarias de forma dinámica.
  • Locales y Empleados: Datos del restaurante físico y sus trabajadores. Cada local tiene su propia capacidad que la IA consulta para verificar la disponibilidad.
  • Clientes: Se almacena su DNI, Nombre, Apellido y Teléfono para la gestión de sus perfiles. Cuando un cliente nuevo realiza una reserva, la IA solicita su DNI y crea automáticamente su perfil en la base de datos.
  • Reservas: Almacena la Fecha, ID del local y del cliente. Durante las fases de prueba se detectó la necesidad de expandir las capacidades del agente: se añadió una tabla intermedia para registrar el menú que el cliente ha pedido anticipadamente en su reserva, si es que lo solicita.

Evolución del Esquema

Durante las fases de prueba se detectó la necesidad de hacer múltiples retoques para expandir las capacidades del agente y crear una base de datos más cómoda y realista. Por ejemplo, en reservas se terminó considerando que era necesario añadir una tabla con el menú que han pedido en la reserva si es que han solicitado uno. Esto llevó al desarrollo de tablas nuevas para poder tener en presencia pedidos especiales: donde el usuario únicamente pide un plato individual en lugar de un simple menú, o un usuario pidiendo múltiples menús para una misma reserva. El esquema final cuenta con un total de 12 tablas interrelacionadas.

Memoria a Largo Plazo

Para evitar que el agente "olvidara" de qué estaba hablando el usuario entre mensaje y mensaje, se configuró un módulo de memoria persistente conectado a PostgreSQL. El sistema logra leer y escribir el historial conversacional exitosamente utilizando el identificador único del chat de Telegram, dotando al sistema de una naturalidad y continuidad imprescindibles para una atención al cliente de calidad.

-- Ejemplo de consulta de IA para información dinámica
SELECT p.nombre, a.tipo_alergeno
FROM comida p
JOIN alergenos a ON p.id_alergeno = a.id
WHERE p.dia_s_disp = true;

Gestión y Agendamiento

Compuesto por acciones estructuradas por "campos", permite a la IA interactuar directamente con la agenda del restaurante empleando la API de Google Cloud Console. El módulo soporta dos operaciones principales: "get events" para consultar la disponibilidad y "create events" para programar nuevas reservas.

Flujo Funcional de una Reserva

  1. El cliente solicita una reserva por Telegram, a menudo en lenguaje natural poco específico.
  2. La IA comprueba en PostgreSQL si el cliente está registrado. En caso contrario, inicia la creación de un perfil pidiendo su DNI y datos personales.
  3. Una vez validados los datos del cliente, la IA ejecuta la acción "get events" en Google Calendar para revisar la disponibilidad del local en la franja horaria solicitada.
  4. Si hay hueco, utiliza el módulo "create events" para asignar el título, hora de inicio y finalización directamente en el calendario del restaurante, evitando solapamientos de forma autónoma.
  5. Simultáneamente, la IA realiza los cambios necesarios en PostgreSQL.
  6. Finalmente, el sistema confirma la reserva al cliente vía Telegram de forma totalmente autónoma.

Validación Autónoma

El agente demostró ser capaz no solo de crear el evento, sino de consultar previamente la disponibilidad mediante "get events", logrando evitar solapamientos de reservas. Además, supo diferenciar entre pedidos imposibles y consultas legítimas, pidiendo aclaraciones al usuario cuando la información era ambigua o insuficiente, creando y editando reservas sin mucha dificultad.

Motor LLM: Devstral latest

Nodo principal que ajusta el comportamiento del "agente". Interpreta el contexto, decide qué estilo de respuesta ofrecer y determina si debe utilizar alguna herramienta externa para cumplir con la petición del usuario. El modelo que impulsa el agente conversacional es Devstral latest de Mistral AI.

La Migración: De Gemini a Mistral

Durante la fase inicial de desarrollo del proyecto, se utilizó como principal proveedor de modelos de inteligencia artificial la API de Gemini, desarrollada por Google. En un primer momento, esta solución parecía ser la opción más adecuada debido a su alto rendimiento y capacidades avanzadas de procesamiento de lenguaje natural.

Sin embargo, a medida que avanzaba el desarrollo y aumentaba la demanda global del servicio externo, se comenzaron a experimentar limitaciones significativas derivadas de la alta carga sobre la API. El elevado número de solicitudes concurrentes provocaba tiempos de respuesta inestables, mayores latencias y, en determinados casos, errores en las peticiones.

Tras identificar este problema, se evaluaron alternativas que permitieran mantener la funcionalidad del sistema con mayor estabilidad y control sobre los recursos. Finalmente, se optó por integrar los modelos de Mistral AI, una empresa europea especializada en modelos de lenguaje. Devstral latest ofrece un modelo altamente avanzado con una API de cuota gratuita generosa, siendo capaz de resolver peticiones complejas que modelos más básicos no lograban procesar correctamente. Además, es uno de los modelos más rápidos del mercado, lo que resulta adecuado para un proyecto que requiere una conversación fluida.

Resultado de la migración: Se observó una reducción considerable en los tiempos de respuesta, así como una disminución notable en la tasa de errores asociados a las solicitudes a la API. Esta decisión contribuyó a aumentar la fiabilidad del sistema durante el proceso de desarrollo y pruebas.

Prompt Engineering y Seguridad

Para que el modelo de inteligencia artificial logre cumplir su objetivo, se ha estado puliendo a lo largo del curso un prompting que constituye el núcleo de sus objetivos como asistente. Este system prompt enseña al agente la estructura interna de la base de datos, cómo utilizar las herramientas de Calendar y finalmente cómo proceder al atender a usuarios. Un aspecto fundamental ha sido la resistencia anti intentos de Jailbreak, garantizando que el agente no se desvíe de su rol de recepcionista del restaurante.

Uso de Herramientas

La capacidad clave de Devstral en este proyecto es que formula respuestas naturales a la par que determina, de forma totalmente autónoma, si debe invocar herramientas externas. Si el agente detecta que necesita accionar algo, llama de forma transparente a los submódulos correspondientes sin intervención humana.

Módulo de Audio STT: Gemini 3.1 Flash Lite

A pesar de haber migrado el razonamiento principal a Mistral, se mantiene Gemini 3.1 Flash Lite de forma exclusiva para la transcripción de audio, aprovechando sus capacidades multimodales. Es un modelo ligero optimizado para esta tarea específica.

Integración de Notas de Voz

El cliente no está obligado a escribir texto. Si envía una nota de voz por Telegram, el orquestador n8n captura el archivo de audio. Gemini 3.1 Flash Lite procesa este audio, lo convierte a texto plano, y ese texto se inyecta directamente al agente de Devstral para que procese la petición y continúe el flujo de trabajo estándar.

Flujo de Procesamiento Dual

Cuando el nodo de Telegram recibe un mensaje, se pasa a un nodo condicional "if" que diferencia automáticamente entre mensajes de voz y de texto. Si es texto, se procesa directamente por el AI Agent. Si es audio, primero se envía a Gemini 3.1 Flash Lite para su transcripción, y luego el texto resultante se inyecta al agente. Esto permite cumplir con uno de los requisitos secundarios: el soporte de Speech-to-Text para una experiencia de usuario completa y natural.

Módulo TTS: ElevenLabs

Como complemento al módulo de transcripción de voz, AITalk incorpora ElevenLabs como motor de conversión de texto a voz. Esto permite que el agente no solo reciba mensajes de voz, sino que también pueda responder de forma hablada.

Respuesta en Voz Natural

El sistema determina el formato de respuesta en función del tipo de entrada del usuario. Si el cliente envió una nota de voz, la IA genera una respuesta que se convierte a audio mediante la API de ElevenLabs antes de ser enviada al chat de Telegram. Si el cliente escribió texto, la respuesta se envía en formato texto estándar. Esta lógica garantiza una experiencia de conversación bidireccional natural y coherente con el medio elegido por el usuario.

Integración en el Flujo

ElevenLabs se integra como un nodo adicional en el flujo de n8n, situado después del AI Agent y antes del nodo de respuesta de Telegram. Cuando el flujo detecta que la entrada original fue un mensaje de voz, la respuesta generada por Devstral se enruta a través del módulo TTS de ElevenLabs, que devuelve un archivo de audio listo para ser enviado como nota de voz al usuario.

Consulta Dinámica de Menú e Información

Una de las funcionalidades principales de AITalk es la capacidad de responder en tiempo real a las consultas de los clientes sobre el menú del restaurante, ingredientes, alérgenos y cualquier información relevante del establecimiento.

Flujo de Consulta

  1. El cliente escribe por Telegram preguntando por el menú o por restricciones alimentarias.
  2. La IA recibe la consulta y, de forma autónoma, decide invocar la herramienta de PostgreSQL para acceder a la información de los platos y alérgenos.
  3. Ejecuta una consulta dinámica en la base de datos que filtra los resultados según los criterios del usuario.
  4. Genera una respuesta coherente e inmediata en lenguaje natural que resuelve la duda del cliente, indicando platos disponibles, ingredientes y alérgenos asociados.

Ejemplo de Interacción

Si un cliente pregunta "¿Qué platos tienen para alguien con alergia a los frutos secos?", la IA consulta las tablas de Comida, Ingredientes y Alérgenos en PostgreSQL, filtra los resultados excluyendo los platos que contienen frutos secos, y responde con una lista de opciones seguras para el cliente. Todo este proceso ocurre de forma transparente y en tiempo real, sin intervención humana.

Gestión y Agendamiento de Reservas

La funcionalidad de gestión de reservas es el núcleo operativo de AITalk. Permite que un cliente realice, consulte y modifique reservas de forma completamente autónoma a través de una conversación en lenguaje natural con el agente de IA.

Proceso Completo de Reserva

  1. Solicitud del cliente: El cliente solicita una reserva por Telegram, proporcionando los datos necesarios de forma natural, sin necesidad de rellenar formularios ni seguir un formato estricto.
  2. Verificación del cliente: La IA comprueba que el cliente esté registrado en la base de datos PostgreSQL. En caso contrario, le crea un perfil pidiendo su DNI y datos de contacto, almacenándolo automáticamente.
  3. Validación de datos: La IA extrae las entidades del mensaje y valida que los datos sean coherentes. Si el usuario proporciona información ambigua o un horario imposible, el agente solicita aclaraciones.
  4. Consulta de disponibilidad: La IA ejecuta la acción "get events" en Google Calendar para revisar la disponibilidad del local en la franja horaria solicitada, evitando solapamientos.
  5. Creación del evento: Si hay disponibilidad, utiliza "create events" para crear el evento en el calendario del restaurante y, simultáneamente, realiza los cambios necesarios en PostgreSQL.
  6. Confirmación: El sistema confirma la reserva al cliente vía Telegram de forma totalmente autónoma, proporcionando un resumen con los detalles de la reserva.

Gestión de Pedidos Especiales

Durante las fases de prueba, se detectó la necesidad de soportar escenarios complejos como: un cliente que pide un plato individual en lugar de un menú completo, un usuario que solicita múltiples menús para una misma reserva, o pedidos que combinan menús y platos a la carta. La base de datos se expandió con tablas intermedias para gestionar estos casos, y la IA fue configurada para manejarlos de forma natural.

Web del Proyecto

El módulo web no es solo un portal corporativo al uso, sino que funciona como la verdadera cara visible de AITalk, adoptando toda la estética y narrativa de un producto SaaS moderno.

Landing Page

La landing page principal está diseñada para aterrizar el concepto y "vender" la visión del proyecto: enganchar al usuario desde el primer vistazo y demostrar cómo la automatización impulsada por IA puede fulminar el caos de las reservas manuales. Es el punto de entrada que traduce toda la complejidad del backend orquestado en una promesa clara de eficiencia para cualquier negocio de hostelería.

Estructura de Pricing

Para darle al proyecto ese toque de realidad y demostrar que tiene recorrido comercial, montamos una fachada completa con una estructura de pricing ficticia, dividida en niveles como "Local" y "Franquicia". Aunque ahora mismo operen como un mockup de cara a la galería, estos planes de suscripción reflejan perfectamente cómo variaría la arquitectura técnica en el mundo real: desde la asignación de memoria compartida y conexiones a bases de datos independientes en PostgreSQL, hasta los límites en la cuota de peticiones al modelo de Devstral. Es, en esencia, la prueba de que la infraestructura soporta un modelo de negocio escalable.

Demo: Restaurantes Manolo

Para demostrar que el despliegue es viable, creamos la demo de "Restaurantes Manolo", un entorno simulado de un cliente final donde llevamos nuestra solución a la práctica. En esta subpágina, la tecnología de AITalk se materializa en un widget flotante inyectado directamente en la interfaz del restaurante. Cualquiera que navegue por la web puede desplegar el chat y comprobar de primera mano cómo el agente comprende el lenguaje natural.

Dashboard de Desarrollo

Como complemento fundamental a la experiencia del cliente, el módulo web incorpora un "Dashboard de Desarrollo" protegido por contraseña. Este panel interno opera como el centro de mando del restaurante y es donde el dueño puede ver las reservas que se han hecho en sus restaurantes y toda la información asociada a estas.

Stack Técnico del Dashboard

  • Backend PHP: Utiliza un backend estructurado en PHP que se conecta de forma segura a PostgreSQL mediante la extensión PDO, garantizando la integridad y seguridad de las consultas.
  • Frontend Asíncrono: En el lado del cliente, se ha implementado un sistema de peticiones asíncronas en JavaScript, lo que permite que la tabla de gestión se actualice en tiempo real de forma automática y sin recargar la página, evitando parpadeos visuales en la interfaz.
  • Acceso Protegido: El dashboard requiere autenticación mediante credenciales de administrador, garantizando que solo el personal autorizado pueda acceder a los datos de reservas y clientes.

Funcionalidades del Panel

El dashboard muestra en tiempo real todas las reservas realizadas, incluyendo información detallada como: identificador de reserva, fecha y hora, nombre y apellidos del cliente, teléfono, local asignado, menús y platos solicitados, número de personas, y estado de la reserva. Los datos se actualizan automáticamente sin necesidad de recargar la página.

Metodología, Viabilidad y Herramientas

Desarrollo Ágil

El equipo ha seguido la metodología Scrum con sprints semanales. Para la organización y reparto de tareas se ha utilizado Taiga como herramienta Kanban, permitiendo visibilizar el estado: To Do, In Progress y Done de cada fase. Aprendiendo simultáneamente el uso de esta herramienta de gestión de tiempo, llegando a bocetos funcionales para el seguimiento del proyecto.

El control de versiones y el trabajo colaborativo del apartado web y los JSON de n8n se ha centralizado en GitHub. Ante los problemas por ejemplo, limitaciones de entornos virtuales con Ubuntu o VirtualBox, la metodología ágil permitió pivotar rápidamente y redistribuir tareas para no bloquear el desarrollo.

Estudio Económico y Viabilidad

El coste del proyecto se sustenta principalmente en las horas de desarrollo del equipo. A nivel de software, se ha priorizado estratégicamente el uso de herramientas gratuitas o de código abierto para asegurar la viabilidad comercial para una PYME:

  • n8n: Versión self-hosted "gratuita" en local.
  • Gemini 3 Flash Lite Preview: Modelo multimodal de Google utilizado a través de API con cuotas gratuitas durante el desarrollo.
  • Devstral latest: Modelo open-weight accesible vía Mistral Studio con API de cuota gratuita generosa, ofreciendo mayor capacidad operativa dentro de los límites gratuitos.
  • PostgreSQL: Software Open Source sin coste de licencia.
  • Servidor: Alojamiento en red local / máquinas virtuales con Ubuntu 24.04.

Infraestructura y Servidor Web

El despliegue se ha planificado en una red local / máquinas virtuales con el sistema operativo Ubuntu 24.04. Ante problemas de limitaciones en entornos virtuales, la metodología ágil permitió pivotar rápidamente sin bloquear el desarrollo.

Nginx como Proxy Inverso

El tráfico de red y la exposición segura de los webhooks generados por n8n hacia Telegram se gestionan utilizando Nginx como proxy inverso, garantizando seguridad y manejo eficiente de peticiones HTTP/HTTPS.

Seguridad: HTTPS con SSL/TLS

Toda la arquitectura funciona bajo el protocolo HTTPS mediante certificados SSL/TLS. Dado que el sistema maneja credenciales de acceso de administradores y datos personales de los clientes en las reservas, la encriptación resulta un requisito fundamental.

Protocolo HTTP/3 y QUIC

La red se ha creado usando el estándar HTTP/3 basado en el protocolo QUIC. Esta mejora en la infraestructura optimiza las peticiones asíncronas del Dashboard, asegurando que el flujo de datos en tiempo real entre PostgreSQL, PHP y el navegador sea ultrarrápido y ofrezca la experiencia impecable que exige un producto profesional. QUIC es un protocolo de transporte encriptado por defecto, diseñado para proporcionar conexiones más rápidas y seguras, sirviendo como fundamento subyacente para las nuevas generaciones de transferencia web asíncrona.
