Arquitectura del Sistema IA Agéntica

AITalk es un sistema automatizado e inteligente de atención al cliente diseñado para cubrir las necesidades de empresas de hostelería, permitiendo gestionar reservas, responder consultas sobre el menú y mantener conversaciones fluidas en lenguaje natural.

Base de Datos (PostgreSQL)

Almacena el inventario de menús, información de alérgenos, locales, y la memoria contextual de las conversaciones con los clientes.

Módulo Telegram (Frontend)

Actúa como frontend principal entre el cliente y el sistema. Escucha peticiones y retorna las respuestas generadas por la IA directamente al chat del usuario, ya sea en su dispositivo móvil o de escritorio.

¿Por qué Telegram?

Al delegar la tarea de atención al cliente a un agente de IA accesible mediante una plataforma de uso masivo como Telegram, se mejora significativamente la experiencia del usuario final. El cliente no tiene que lidiar con interfaces web complejas ni sufrir tiempos de espera telefónicos. Basados en nuestra experiencia técnica, Telegram resultó ser la opción más robusta y versátil para esta tarea frente a otras alternativas de mensajería.

Orquestación con n8n

n8n es el pilar fundamental y orquestador del proyecto. Permite integrar y coordinar los diferentes servicios que forman parte de la solución a través de flujos de trabajo automatizados, gestionando la comunicación entre la IA, PostgreSQL, Google Calendar y Telegram.

Arquitectura Modular

La arquitectura sigue un modelo de nodos modulares centralizados. El sistema se divide en piezas más pequeñas que, al unirse, conforman un sistema robusto:

  • Trigger: El nodo de Telegram escucha continuamente mensajes nuevos.
  • Agente IA: El mensaje se pasa al nodo de "AI Agent" configurado con el modelo Devstral y un rol específico predefinido.
  • Memoria en Postgres: Se utiliza un módulo de memoria permanente almacenando en la base de datos, lo que permite que la IA recuerde el contexto a largo plazo.
  • Tools (Herramientas): Si el agente detecta que necesita accionar algo, llama de forma transparente a los submódulos (Calendar, BBDD).
  • Respuesta: El agente formula una respuesta natural, ejecuta las acciones y n8n la devuelve al bot de Telegram.

Base de Datos (PostgreSQL)

Conexión directa a la base de datos relacional del restaurante. Hemos elegido PostgreSQL por ser un software Open Source robusto y sin coste de licencia. Además de almacenar el inventario, sirve para guardar el contexto (memoria) de la IA.

Esquema Conceptual Completo

Durante el desarrollo, diseñamos un esquema relacional profundo con las siguientes entidades principales:

  • Comida e Ingredientes: Tablas relacionadas donde un Menú incluye platos (Comida) que a su vez contienen Ingredientes. Se controlan los Alérgenos (tipos y gravedad).
  • Locales y Empleados: Datos del restaurante físico (Ciudad, Horario, Capacidad) y sus trabajadores (Nombre, Apellido, Ocupación, Salario).
  • Clientes: Se almacena su DNI, Nombre, Apellido y Teléfono para la gestión de sus perfiles.
  • Reservas: Almacena la Fecha e ID del local y cliente. Nota de desarrollo: Durante las fases de prueba nos dimos cuenta de que debíamos expandir las capacidades; por ello añadimos una tabla intermedia para registrar el "menú" que el cliente ha pedido anticipadamente en su reserva, si es que lo solicita.

WHERE p.dia_s_disp = true;

Gestión y Agendamiento (Google Calendar)

Compuesto por acciones estructuradas por "campos", permite a la IA interactuar directamente con la agenda del restaurante empleando la API de Google Cloud Console.

Flujo Funcional de una Reserva

  1. El cliente solicita una reserva (fecha, hora, personas).
  2. La IA comprueba en PostgreSQL si el cliente está registrado. En caso contrario, inicia la creación de un perfil pidiendo su DNI y datos.
  3. Una vez validados los datos, la IA ejecuta la acción "get events" en Calendar para revisar la disponibilidad del local.
  4. Si hay hueco, utiliza el módulo "create events" para asignar el título, hora de inicio y finalización directamente en el calendario.
  5. Simultáneamente, la IA realiza los cambios en PostgreSQL (añadiendo la reserva a la tabla y vinculando el menú si aplica).
  6. Finalmente, el sistema confirma la reserva al cliente vía Telegram de forma totalmente autónoma.

Motor LLM: Devstral latest (Mistral)

El cerebro del agente conversacional está impulsado por el modelo Devstral latest de Mistral AI.

La Migración: De Gemini a Mistral

Durante la fase inicial del desarrollo, utilizamos la API de Gemini (Google) como proveedor principal por sus capacidades avanzadas. Sin embargo, a medida que aumentaba la demanda global de esa API externa, comenzamos a experimentar limitaciones significativas: tiempos de respuesta inestables, mayores latencias y errores debido a la alta carga.

Tras evaluar alternativas, optamos por integrar los modelos de la empresa europea Mistral AI (vía Mistral Studio). Aunque inicialmente parecía tener un rendimiento ligeramente diferente, su disponibilidad mediante APIs gratuitas generosas y su menor saturación nos permitieron mejorar drásticamente la estabilidad. Resultado de la migración: Reducción considerable en tiempos de respuesta y disminución notable de la tasa de errores.

Uso de Herramientas (Tools)

La capacidad clave de Devstral en este proyecto es que formula respuestas naturales a la par que determina, de forma totalmente autónoma, si debe invocar herramientas externas (ej. consultar Postgres para dudas de alérgenos o ejecutar las consultas de calendario explicadas en la sección anterior).

Módulo de Audio STT: Gemini 3.1 Flash Lite

A pesar de haber migrado el razonamiento principal a Mistral, mantenemos Gemini 3.1 Flash Lite exclusivamente por sus increíbles capacidades multimodales para la transcripción de audio (Speech-to-Text).

Integración de Notas de Voz

El cliente no está obligado a escribir texto. Si envía una nota de voz por Telegram, el orquestador n8n captura el archivo de audio. Al ser un entorno de hostelería, necesitábamos un modelo robusto. Gemini 3.1 Flash Lite procesa este audio a una velocidad extrema, lo convierte a texto plano, y ese texto se inyecta directamente al agente de Devstral para que procese la petición y continúe el flujo de trabajo estándar.

Metodología, Viabilidad y Herramientas

Desarrollo Ágil (Scrum)

El equipo ha seguido la metodología Scrum con sprints semanales. Para el reparto de tareas se ha utilizado Taiga (herramienta Kanban), visibilizando el estado en To Do, In Progress, y Done. El trabajo colaborativo y control de versiones de código y de los JSON de n8n se centraliza en GitHub.

Estudio Económico y Viabilidad

El coste del proyecto se sustenta principalmente en nuestras horas de desarrollo. A nivel de software, se ha priorizado estratégicamente el uso de herramientas gratuitas o de código abierto (Open Source) para asegurar la viabilidad comercial para una PYME. Utilizamos la versión self-hosted de n8n, base de datos PostgreSQL gratuita, y modelos de IA optimizados que ofrecen cuotas gratuitas generosas (Devstral API).

Infraestructura y Servidor Web

El despliegue se ha planificado en una red local / máquinas virtuales con el sistema operativo Ubuntu 24.04. Ante problemas de limitaciones en entornos virtuales (ej. VirtualBox), la metodología ágil nos permitió pivotar rápidamente sin bloquear el desarrollo.

El tráfico de red y la exposición segura de los webhooks generados por n8n hacia Telegram se gestionan utilizando Nginx como proxy inverso, garantizando seguridad y manejo eficiente de peticiones HTTP/HTTPS.
